Advanced Tab Usage

The tabbed pane on the right side of QemuJuicys has a tab "Advanced".

If using a generic VM in a usual way only, one might ignore this tab, but QemuJuicys "Advanced" tab offers several possibilities.

It has two radio buttons and two text areas. The default "Add special QEMU parameters" button displays in the upper area all QEMU parameters created automatically by QemuJuicy, taking all defined VM properties. This text area cannot be changed or edited, but copied into the clipboard for further usage. Missing or special QEMU parameters may be added within the lower text area, using the same style as above.
Clicking the button "Full QEMU definitions" changes this behaviour. Now the user is completely responsible for the upper text area, regardless of the settings of VM properties (they are ignored). Any parameters from the lower text area are added, if the VM is started. Changing the radio buttons will not change the "Full QEMU definitions" content, it is saved and survives a QemuJuicy exit.

This offers some possibilities:

  • Add QEMU parameters not known to QemuJuicy. And yes, there are many of them, take a look at "www.qemu.org" -> main documentation section.
  • To add special parameter options by setting a VM property to "None/Advanced Tab" and adding the specialiced, more complex parameter option within the lower text area.
  • Set up a VM, use the "Copy" button or context menu to use the complete text for further use (on a CLI/shell).
  • Set up a VM, use the "Store" button to write the QEMU call of this VM to a file. This file - made executable - may be used as a starter for VMs, QemuJuicy doesn't need to run. Complex networking between the VMs (some tens of them) might find this helpful, or a shell or *.bat file can be used as a menu entry to start a VM.
  • In small to medium VM environments (some tens of VMs, several servers), QemuJuicy can be used to develop, test and modify the VMs. Use the "Store" button to save the QEMU call, copy the shell/batch file together with the virtual harddisk to the server and start the (now remote) VM on that server (possibly using ssh, sshfs). Simple, quick solution, no overhead, easy to handle, secure: yes, all components could be Open Source.
  • Copying VMs with specialized parameters, including their virtual harddisks.
  • Varying parameters in a GUI for testing, using QemuJuicy as a starter.
  • Using QemuJuicy just as a starter, setting the complete QEMU command with "Full QEMU definitions". This could be used for rare VMs, like FPGA machines, having one place (QemuJuicy) to manage all VMs.
